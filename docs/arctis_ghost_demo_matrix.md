# Arctis Ghost — Demo-Matrix (Landingpage & Story-Arc)

**Zweck:** Mehrere Demos — **kuratiert**, nicht überladen. Jedes Modul trifft einen **anderen Nerv**; zusammen erzählen sie **eine** Geschichte:

> **Arctis Ghost ist kein Tool — es ist ein neues Interface-Paradigma** (Dateisystem, HTTP, Evidence, keine zweite Governance-Wahrheit).

## 1. Die sechs Erlebnis-Module (Demo-Matrix)

Jedes Modul hat **einen Namen**, **eine Zielgruppe**, **eine Kernbotschaft** und **sichtbare Bausteine**. Features ohne Produktreife kennzeichnen (z. B. „Roadmap“ / „teilweise“) — Vertrauen schlägt Marketing.

### Modul 1 — Entscheider (C-Level)

| Feld | Inhalt |
|------|--------|
| **Name** | *Von Datei zu Audit in 10 Sekunden* |
| **Zielgruppe** | CEO, CFO, COO |
| **Botschaft** | Spart Geld, schafft Vertrauen, ist **sofort** einsatzbereit — ohne Großprojekt. |
| **Inhalt (Show)** | Datei in Ordner → Ergebnis → **Audit-Stamp** (sichtbar) → **Profit-/Savings-Hinweis** (`__SAVINGS_REPORT` / Insights) → **Evidence-Bundle** öffnen |
| **Warum** | C-Level will **Impact**, keine Architekturfolien. |
| **SEO / Hook (Beispiele)** | „KI Audit ROI“, „Invoice governance demo“ |

### Modul 2 — IT-Security & Compliance

| Feld | Inhalt |
|------|--------|
| **Name** | *Governance ohne Schatten-IT* |
| **Zielgruppe** | CISO, Datenschutz, Legal |
| **Botschaft** | **Keine zweite Engine**, keine Policy-Duplikation, **volle Nachvollziehbarkeit** über die API. |
| **Inhalt (Show)** | `skill_reports` → Policy-/Config-**Diff** (wo verfügbar) → **Evidence-Subset** → **PII Local-Relay** (Horizont; klar als Pilot labeln) → **Verify-Flow** (nur wenn Endpoint echt existiert) |
| **Warum** | Diese Gruppe **freigibt** oder blockiert. |
| **SEO / Hook** | „AI governance ohne Shadow IT“, „policy diff demo“ |

### Modul 3 — Entwickler & Tech-Teams

| Feld | Inhalt |
|------|--------|
| **Name** | *API-First: Ghost als dünnster Client* |
| **Zielgruppe** | Entwickler, Architektinnen |
| **Botschaft** | **Kein** Pflicht-UI, **kein** Framework-Zwang — **HTTP + Ordner**. |
| **Inhalt (Show)** | Request/Response (OpenAPI) → **Idempotency-Key** → **Skill-Envelope** → **`ghost doctor`** (live) → **`ghost explain RUN_ID`** (Kurzüberblick) |
| **Warum** | Tech muss sehen: Ghost **addiert** keine zweite Komplexitätsschicht. |

## 2. Landingpage-Struktur: „Wähle deine Demo“

**Sektionstitel (Vorschlag):** *Welche Frage hast du?*

Jede **Kachel** = ein Modul (1–6). Darunter **eine** Leitfrage in Alltagssprache — Besucher erkennen sich ohne Jargon.

| Kachel / Modul | Leitfrage (Headline) |
|----------------|----------------------|
| 1 C-Level | *Wie viel Geld spare ich — und wie schnell sehe ich Vertrauen?* |
| 2 Security | *Wie sicher ist das — ohne Schatten-IT?* |
| 3 Tech | *Wie funktioniert das technisch — wirklich minimal?* |
| 4 Sachbearbeitung | *Wie benutze ich das — ohne Schulung?* |
| 5 Innovation | *Was ist das Neue — jenseits von Chatbots?* |
| 6 Audit | *Wie prüfe ich alte Systeme und Altlasten?* |

**Pro Kachel drei Einstiege (gleiche Story, steigende Tiefe):**

| Format | Dauer | Zweck |
|--------|--------|--------|
| **Kurz-Clip** | ca. 20 s | Hook, Autoplay-freundlich, Social |
| **Demo 60** | ca. 60 s | Siehe [arctis_ghost_demo_60.md](arctis_ghost_demo_60.md), Inhalt **modulspezifisch** trimmen |
| **Deep-Dive** | ca. 3 min | Terminal + Ordner + ein API-Call / Evidence-Detail |

So bleibt die Seite **klickbar und sortiert**, nicht eine flache Feature-Liste.

## 3. Story-Arc über alle Module (ein roter Faden)

1. **Paradigma:** Arbeit passiert dort, wo Dateien ohnehin leben — Ghost ist **Kanal**, nicht Portal.  
2. **Vertrauen:** Entscheidungen kommen aus **Arctis API** (Policy, Budget, Evidence) — Ghost **dupliziert** sie nicht.  
3. **Rollen:** Jede Rolle sieht **ihre** Sprache (ROI, Risiko, API, Einfachheit, Vision, Revision).  
4. **CTA:** Immer derselbe nächste Schritt: *Demo-Ordner / API-Key / Sandbox* — ein Funnel, sechs Einstiege.

---

## 4. Warum mehrere Demos (kurz, bewusst)

- **Unterschiedliche Fragen:** CFO ≠ CISO ≠ Entwicklerin ≠ Sachbearbeiterin.  
- **Eigener PLG-Funnel pro Modul:** anderes „Aha“, gleiche technische Basis.  
- **SEO:** je Modul eigene Landing-URL oder Anchor (`#demo-security`, …).  
- **Sales:** im Gespräch **einen** Link schicken — passend zur Person.

---

## 5. Kuratier-Regeln (Anti-Overkill)

- Maximal **sechs** öffentliche Module; alles Weitere unter „Roadmap“ oder Blog.  
- Pro Modul **ein** Held-Use-Case (z. B. Rechnung, Vertrag, Logfile) — nicht drei.  
- Nicht jedes Horizont-H3-Feature in jeder Demo zeigen; **Ehrlichkeit** bei noch nicht gebauten Punkten.  
- **Demo 60**-Skripte pro Modul: nur die **Abschnitte 3–4** aus [arctis_ghost_demo_60.md](arctis_ghost_demo_60.md) mit modulspezifischen Artefakten tauschen.
